> Obsidian으로 이관한 뒤, 실제 사용에 맞게 폴더 노트와 정렬 방식을 맞춘 기록

Notion에서 Obsidian으로 넘어온 이유는 이전 글에 따로 정리했다. 이 글은 그 다음 단계에 가깝다.

Obsidian으로 넘어온 뒤 실제로 쓰려면 노트 구조를 먼저 정해야 했다. 파일만 계속 쌓이면 시간이 지날수록 찾기 어렵고, 나중에 블로그 글로 옮길 때도 정리가 번거로워진다.

그래서 폴더 노트를 기본 단위로 잡고, 첨부 파일 위치와 파일 탐색기 정렬 방식까지 같이 맞췄다. 이 글에서는 동기화 설정이 아니라, Obsidian 안에서 노트를 어떤 구조로 관리할지에만 집중한다.

## 1. 구성 방향

전체 방향은 단순하다.

- 노트는 폴더 노트 구조로 관리한다.
- 첨부 파일은 각 노트의 `attachments` 폴더에 둔다.
- 상위 노트의 링크 순서를 하위 노트 정렬 순서로 사용한다.
- 정렬 파일은 자동으로 갱신한다.

노트는 오래 쌓일수록 처음 정한 구조의 영향을 크게 받는다. 그래서 처음에는 조금 번거롭더라도 구조를 일관되게 가져가는 편이 낫다고 판단했다.

## 2. 폴더 노트 구조

노트는 가능한 한 아래 형태로 만든다.

```text
페이지명/
├── 페이지명.md
└── attachments/
```

예를 들어 `홈서버`라는 주제가 있으면 `홈서버.md` 파일만 하나 두는 것이 아니라, `홈서버/홈서버.md` 형태로 둔다. 이미지나 캡처 파일은 같은 폴더의 `attachments` 아래에 저장한다.

이렇게 하면 한 주제에 대한 본문과 첨부 파일을 한 단위로 다룰 수 있다. 나중에 글로 옮기거나, 다른 주제로 분리하거나, 오래된 자료를 정리할 때도 판단 기준이 명확해진다.

## 3. 사용하는 플러그인

현재 구조에서 핵심이 되는 플러그인은 아래와 같다.

| 플러그인 | 용도 |
| --- | --- |
| Folder Notes | 폴더와 같은 이름의 대표 노트를 사용 |
| Templater | 새 폴더 노트 생성 자동화 |
| Custom File Explorer Sorting | `sortspec.md` 기준으로 파일 탐색기 정렬 |
| Dataview | 메타데이터 기반 목록이나 대시보드 작성 |
| Importer | 외부 자료를 Obsidian으로 가져올 때 사용 |

목차는 별도 플러그인을 추가하지 않고, Obsidian 기본 기능인 개요(Outline)를 먼저 사용하기로 했다. 데스크톱에서는 떠 있는 목차가 편할 수 있지만, 모바일에서는 화면을 가릴 가능성이 있어서 기본 기능으로 시작하는 편이 더 안정적이다.

## 4. 새 폴더 노트 생성

새 노트를 만들 때마다 폴더, 본문 파일, 첨부 파일 폴더를 직접 만드는 것은 번거롭다. 그래서 Templater로 새 폴더 노트 생성 템플릿을 만들어 사용한다.

동작 흐름은 아래와 같다.

1. 새 노트 이름 입력
2. 현재 위치에 같은 이름의 폴더 생성
3. 폴더 안에 같은 이름의 `.md` 파일 생성
4. `attachments` 폴더 생성
5. 현재 노트에 새 노트 wikilink 삽입

결과적으로 아래 구조가 자동으로 만들어진다.

```text
입력한 이름/
├── 입력한 이름.md
└── attachments/
```

이 방식의 장점은 상위 노트에서 하위 노트 링크를 먼저 만들고, 그 링크 순서를 그대로 정렬 기준으로 사용할 수 있다는 점이다.

## 5. 상위 노트의 링크 순서를 정렬 기준으로 사용

Obsidian에서 파일 탐색기 순서를 직접 관리하다 보면, 실제 글의 흐름과 파일 목록의 순서가 어긋나는 경우가 많다.

그래서 상위 노트에 적은 wikilink 순서를 기준으로 하위 노트의 탐색기 순서를 맞추도록 했다.

예를 들어 상위 노트에 아래처럼 적으면,

```markdown
[[첫 번째 글]]

[[두 번째 글]]

[[세 번째 글]]
```

하위 폴더의 `sortspec.md`도 같은 순서로 만들어진다.

```yaml
---
sorting-spec: |
  첫 번째 글
  두 번째 글
  세 번째 글
---
```

이렇게 하면 문서 안에서 보는 목차 순서와 파일 탐색기에서 보는 순서가 같아진다.

## 6. sortspec.md 자동 갱신

Custom File Explorer Sorting은 `sortspec.md`를 기준으로 파일 탐색기 순서를 바꿔준다. 문제는 이 파일을 매번 직접 관리하기 어렵다는 점이다.

그래서 서버에서 작은 자동화 스크립트를 돌려 폴더 노트가 수정될 때마다 `sortspec.md`를 자동으로 갱신하도록 했다.

동작 방식은 아래와 같다.

- 일정 주기로 vault의 폴더 노트를 확인한다.
- 폴더 이름과 같은 이름의 `.md` 파일을 폴더 노트로 판단한다.
- 폴더 노트 안의 wikilink를 읽는다.
- 실제 존재하는 하위 노트나 폴더만 정렬 대상에 넣는다.
- 링크에 없는 하위 항목은 알파벳순으로 뒤에 붙인다.
- 정렬할 하위 항목이 없으면 빈 `sortspec.md`를 남기지 않는다.

정렬 파일은 아래 형식을 사용한다.

```yaml
---
sorting-spec: |
  항목1
  항목2
---
```

`sorting-spec: |`만 있고 아래 항목이 비어 있으면 플러그인에서 파싱 문제가 생길 수 있다. 그래서 하위 항목이 없는 폴더에는 `sortspec.md`를 만들지 않는 쪽으로 정리했다.

## 7. 정리 기준

Obsidian에는 정리된 글뿐 아니라 임시 메모, 복사해온 자료, 나중에 볼 링크도 같이 들어온다. 다만 모든 자료를 오래 보관할 필요는 없다.

복사해온 자료나 오래된 도구에 묶인 자료가 현재 작업 흐름과 맞지 않으면 정리 대상이다. 특히 다시 볼 가능성이 낮고 직접 정리한 내용도 아니라면 따로 보관하기보다 삭제하는 편이 낫다.

반대로 직접 작업하면서 얻은 설정, 문제 해결 과정, 운영 기준은 최대한 설명형 글로 남긴다. 나중에 블로그 글로 옮길 수 있도록 처음부터 공개 가능한 문장 구조에 가깝게 작성하는 편이 좋다.

## 8. 정리

이번 설정의 핵심은 Obsidian을 단순히 파일을 저장하는 공간으로 두지 않고, 오래 쓸 수 있는 노트 구조로 맞추는 것이다.

폴더 노트로 주제 단위를 만들고, 상위 노트의 링크 순서를 정렬 기준으로 사용하고, 반복되는 정렬 파일 관리는 자동화했다.

아직 완성된 구조라기보다는 계속 다듬어가는 운영 방식에 가깝다. 그래도 폴더 구조, 첨부 파일 위치, 정렬 기준이 정해져 있으니 노트가 늘어나도 덜 무너지는 형태가 됐다.
